# Froggy on RAG v1.2.0 — Branding, per-request Froggy RAG controls, vector chunk filters, indexing yields, docs refresh, and PNG IHDR splash sizing

**Release tag:** `v1.2.0`  
**Package version:** 1.2.0  
**Type:** Minor release (product, passthrough / retrieval, indexing UX, documentation, startup polish)

**Baseline:** Changes since **v1.0.18** are included in **v1.1.0** and **v1.2.0** (this document rolls them up for one upgrade jump from **v1.0.18**.)

---

## Summary

Since **v1.0.18**, the product is rebranded to **Froggy on RAG** (desktop title, tray, splash copy, **`electron-builder`** `productName`, README, and distribution docs). **LLM Passthrough** and the **inbound HTTP** chat proxy gained a structured **`froggy`** request object: callers can steer **namespace**, **topK**, **search algorithm**, **RAG on/off**, **`includeMetadata`**, **`includeWebSearch`**, **tag and metadata filters**, **named prompt profiles** with **`variables`**, and **extra instructions**—with normalization, warnings for missing profiles, and optional **`[METADATA]`** sections that summarize requested vs retrieved tags/metadata. **Vector retrieval** now applies **`SearchService.buildChunkFilter`** so tag/metadata constraints filter **chunks** after search (with dedicated **`search-service-filters`** tests). **DocumentProcessor** yields to the event loop during long Markdown and plain-text merge passes so heavy ingestion stays more responsive. The **renderer** and **IPC** layers were expanded to surface the new behavior in **Settings** and passthrough flows. Documentation and in-repo copy prefer **folder** over **directory** where it describes user-visible paths; **`npm test`** runs **`node --test`**. **v1.2.0** finishes the arc with **splash** improvements: splash window bounds come from the PNG **IHDR** via **`readPngDimensionsSync`**, and the splash window **`show()`**s immediately after **`loadFile`** so startup is not blocked on a duplicate decode/CSS background load, with the existing **`ready-to-show`** fallback still avoiding a blank dismiss race.

## Highlights

### Branding and packaging

- **Froggy on RAG** replaces **Froggy RAG MCP** across **`package.json`**, **README**, **docs/**, **splash** / **index** titles, **`scripts/create-dist.js`**, and related UI strings. **`build.productName`** is **Froggy on RAG** for installers and updates.
- **Splash image** asset updated on disk with the rebrand; path remains **`images/Froggy RAG - Splash.png`** in **`electron-builder`** `files`.

### LLM Passthrough and inbound HTTP (`froggy`)

- **`resolveFroggyConfig`** (in **`src/main/services/llm-passthrough.js`**) merges **`froggy`** from the inbound JSON body with header/query-style options, normalizes **`namespace`**, **`topK`** (capped), **`algorithm`** (**`hybrid`**, **`bm25`**, **`tfidf`**, **`vector`**), **`rag`**, **`includeMetadata`**, **`includeWebSearch`**, **`tags`**, **`metadata`**, **`promptProfile`**, **`variables`**, and **`extraInstructions`**, and builds **`retrievalQuery`** (including a line for request tags when present).
- **`normalizeChatMessages`** flattens OpenAI-style multipart **`content`** (concatenates **`text`** parts; non-text parts are skipped for the RAG query path).
- **`formatSearchHitsForContext`** formats hits with source file and **namespace**; omits a noisy raw **`Metadata:`** dump in the default context block.
- **`buildRagMetadataSection`** / **`collectRetrievedMetadata`** / **`normalizeFroggyMetadata`** support the optional **`[METADATA]`** block when **`includeMetadata`** is true.
- **`completeChatProxy`** passes **`filters`** from **`froggy`** into **`searchCorpusInNamespaces`** so retrieval honors the same constraints as the UI/MCP paths.
- **Tests:** **`test/llm-passthrough.test.js`** covers normalization, **`resolveFroggyConfig`**, metadata sections, and related helpers (via **`__testing`** exports).

### Vector search — chunk-level filters

- **`SearchService.buildChunkFilter`** builds a predicate from **`filters.tags`** and **`filters.metadata`**: matches array tags, comma-separated tag strings, metadata values/keys, and **`key=value`** forms stored under metadata (see **`test/search-service-filters.test.js`** for the contract).
- Retrieval paths that accept **`retrievalSettings.filters`** apply the filter after ranking so only matching chunks enter context.

### Indexing / `DocumentProcessor`

- **`CHUNK_YIELD_INTERVAL`** and **`yieldToEventLoop`** (**`setImmediate`**) break up long synchronous loops during **Markdown** subdivision/merge, plain **`_mergeStringsWithOverlap`**, and hierarchical coarse chunks so the **Electron** main thread can process UI and IPC between batches.
- **`_mergeStringsWithOverlap`**, **`_mergeMarkdownStringsWithOverlap`**, and **`_addHierarchicalCoarseChunks`** are **`async`** where needed to await those yields.

### Renderer, IPC, and services

- **`src/renderer/app.js`**, **`index.html`**, and **`styles.css`** — large update set for passthrough / RAG / settings UX aligned with the new **`froggy`** surface and workflows.
- **`src/main/ipc-handlers.js`**, **`preload.js`**, **`rag-service.js`** — extended handlers and service methods for the above (save/load paths, passthrough-related IPC).
- **MCP / admin** — terminology and small behavioral alignment in **`mcp-tool-registry`**, **`mcp-json-rpc-handler`**, **`admin-routes`**, **`corpus-namespace-query`**, **`namespace-scope`** (folder wording and consistency).

### Documentation and developer workflow

- **README** substantially revised (shorter product story, install pointer, usage).
- **docs/DISTRIBUTION.md**, **INDEXING_PERFORMANCE_IMPROVEMENTS.md**, **LLM App Creation Instructions.txt** — terminology (**folder**), structure, and branding updates.
- **`package.json`** **`test`** script: **`node --test`** for the growing **`test/`** suite.
- **`.gitignore`** comment clarifies the **data** folder purpose.
- **Historical `release-notes/v1.0.13.md` / `v1.0.14.md`** — minor wording alignment.

### Splash / main process (v1.2.0)

- **`readPngDimensionsSync`** — reads PNG **IHDR** width/height from the first 24 bytes without decoding pixels; **`getSplashWindowSize()`** prefers it over **`nativeImage.createFromPath`** for sizing, with fallback for non-PNG or invalid headers. Window size is still clamped to the primary display work area (~**92%**).
- **`createSplashWindow()`** calls **`splashWindow.show()`** right after **`loadFile`**; comments document the interaction with **`splash.html`** CSS background loading. **`ready-to-show`** still handles the race where the main window becomes ready before the splash has painted (**100 ms** visible fallback).

## Upgrade notes

- **From v1.0.18:** Install **v1.2.0** (or **v1.1.0** then update) as usual; **settings.json** and indexes remain compatible. New **`froggy`** fields are optional; existing clients that omit them behave like before (defaults from app settings).
- **Inbound HTTP / OpenAI-compatible clients:** To use **tag/metadata filters**, **`includeMetadata`**, **`includeWebSearch`** per request, or **prompt profiles** with **`variables`**, add a **`froggy`** object to the chat JSON body (see tests and **`llm-passthrough.js`** for shapes). **`includeWebSearch`** still requires **Google Custom Search** to be configured under **Settings → Web Search** when you turn it on.
- **Chunk filters:** Stricter retrieval may return fewer hits when filters exclude all chunks for a document; widen tags/metadata if results disappear unexpectedly.
- **Re-ingestion:** Not required for upgrades; re-ingest only if you want chunks rebuilt under the same Markdown strategy as **v1.0.18** plus the new **yield** behavior (output text should be equivalent aside from responsiveness during ingest).
- **Tests:** Run **`npm test`** after pulling.
