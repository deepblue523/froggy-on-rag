# Froggy on RAG v1.2.5 — Strip `froggy` payload sections before upstream passthrough

**Release tag:** `v1.2.5` (set when publishing)  
**Package version:** 1.2.5 (bump with `npm run version:patch` before tag/build)  
**Type:** Patch release (passthrough payload hygiene / upstream compatibility)

**Baseline:** Changes since **v1.2.4** only. For prior context, see **`release-notes/v1.2.4.md`**.

---

## Summary

This release hardens inbound LLM passthrough forwarding by removing any `froggy` sections from outbound upstream request bodies. Froggy-specific request controls are still consumed locally for retrieval and prompt shaping, but they are now guaranteed not to be forwarded to upstream Ollama/OpenAI-compatible APIs.

## Highlights

### Outbound payload sanitization for passthrough

- **`src/main/services/llm-passthrough.js`** — added recursive sanitizer **`stripFroggySections()`**.
- Sanitization is applied immediately before outbound upstream calls in **`completeChatProxy()`** for:
  - Ollama-compatible forwarding (`/api/chat`)
  - OpenAI-compatible forwarding (`/chat/completions`)
- Nested `froggy` keys are removed recursively (including within forwarded `options` objects) so provider payloads contain only upstream-relevant fields.
- Existing Froggy behavior remains intact for local processing:
  - `froggy` options are still read for RAG retrieval settings
  - prompt/profile/tag/metadata controls still influence injected system context

### Test coverage

- **`test/llm-passthrough.test.js`** — added regression test:
  - **`stripFroggySections removes froggy keys recursively`**
- Confirms top-level and nested `froggy` objects are stripped while non-Froggy fields are preserved.

## Upgrade notes

- **Client impact:** Existing clients can continue sending `froggy` controls to inbound passthrough listeners; controls are honored locally but no longer forwarded upstream.
- **Upstream compatibility:** Reduces risk of unknown-field handling issues with strict upstream API implementations.
- **Behavioral scope:** No expected response-format or transport changes; this patch only affects outbound payload shape.
